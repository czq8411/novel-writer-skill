# 模式B-2：细纲工程师

> **所属**：novel-writer 核心能力库（模式B子模块） | **触发词**：`细纲`、`章节规划`、`拆解`
> **全局规则**：遵循主控文件第二章核心铁律

---

## MCP联动
- **Memory**：确认前文伏笔、人物当前状态
- **Sequential Thinking**：推演因果链条，确保逻辑闭环
- **Filesystem**：读取前3-5章细纲（确认剧情连续性）+ 读取大纲文件，按模块优先级提取信息（6.3章节摘要→6.2单元→5.2/5.4人物→7.3伏笔→7.1场景），确保细纲与大纲一致

## 前置要求
细纲是大纲的**执行级拆解**，必须严格对齐大纲模块（6.1全书结构、6.2剧情节点、6.3章节摘要、5.2主角成长、5.4配角弧线、7.3伏笔管理），并确认与前文的人物状态、伏笔状态、剧情逻辑连续性。

**撰写流程**：
1. 读取前3-5章细纲，确认伏笔状态、人物连续性、剧情脉络
2. 按优先级读取大纲：6.3章节摘要 → 6.2单元 → 5.2/5.4人物 → 7.3伏笔 → 7.1场景
3. Sequential Thinking推演因果链
4. 按输出结构逐项填写，每项标注大纲模块编号
5. 一致性自检后通过Filesystem保存

## 一致性检查规则
细纲完成后必须对照大纲进行模块级一致性检查，确保核心情节、爽点类型、人物状态、伏笔编号、场景设计、境界约束、时间线等与大纲各模块完全对齐。

## 变更影响分析
修改大纲/细纲时，必须分析变更对已写章节的影响范围，列出受影响章节清单和修复建议。修改前自动备份原文件。

## 质量检查标准（细纲完成后必须检查）
- [ ] **章节一致性**：细纲核心情节是否与大纲6.3章节摘要完全一致
- [ ] **卷-单元一致**：细纲定位是否与大纲6.1全书结构总览和6.2剧情节点设计一致
- [ ] **爽点一致性**：爽点类型、强度、密度是否符合大纲6.3+6.5+6.6要求
- [ ] **人物一致性**：人物境界、状态、关系是否符合大纲5.2+5.4+5.5的当前阶段
- [ ] **伏笔一致性**：伏笔回收（F编号）和埋下是否符合大纲7.3伏笔管理追踪表
- [ ] **场景一致性**：场景的功能定位和出场人物是否符合大纲7.1特殊场景速查
- [ ] **功法宝物一致**：涉及功法/宝物的获取节点是否符合大纲7.2关键功法/宝物设定
- [ ] **境界一致性**：人物境界是否在3.2等级阶梯和5.2境界轨迹的约束范围内
- [ ] **时间线一致**：是否遵循2.2核心时间线和2.3时间流速体系
- [ ] **引用完整性**：所有大纲引用是否标注了模块编号和子节编号
- [ ] **境界冲突检查**：是否违反一致性检查规则中的境界约束
- [ ] **伏笔编号检查**：新伏笔编号是否与现有编号冲突，已回收编号是否被重复使用
