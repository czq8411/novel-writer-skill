# 模式E子模块：批量自动诊断+自动修复

> 按需加载：涉及批量诊断、逐章自动诊断、自动修复时加载本文件。
> 全局规则：遵循主控文件第二章核心铁律

---

## 触发方式

| 触发词 | 功能 | 示例 |
|--------|------|------|
| `细纲批量诊断` `诊断全部细纲` `自动修复细纲` | 对指定范围的细纲逐章自动诊断+自动修复 | `细纲批量诊断 第1-60章` |
| `正文批量诊断` `诊断全部正文` `自动修复正文` | 对指定范围的正文逐章自动诊断+自动修复 | `正文批量诊断 第1-10章` |
| `批量诊断` `全量诊断` `一键诊断修复` | 同时诊断细纲+正文，含大纲反向补充 | `批量诊断 细纲第1-60章 正文第1-10章` |
| `仅诊断不修复` `只诊断` | 仅诊断不自动修复，输出报告供人工决策 | `细纲批量诊断 第1-60章 仅诊断不修复` |
| `大纲补充` `反向补充大纲` `细纲反馈大纲` | 基于细纲诊断结果识别大纲缺口并生成补充方案 | `大纲补充 基于第1-60章细纲` |
| `自动补充大纲` | 自动识别缺口并直接补充到大纲 | `自动补充大纲 基于全部细纲` |
| `大纲缺口检测` `大纲完整性检查` | 仅检测大纲缺口，不自动补充 | `大纲缺口检测` |

## 功能概述

批量自动诊断+自动修复模式将模式E的14维诊断能力扩展为逐章批量处理，支持：

1. **细纲批量诊断+修复**：对N章细纲逐章执行诊断，P0/P1问题自动修复，P2/P3问题标记建议
2. **正文批量诊断+修复**：对N章正文逐章执行诊断，P0/P1问题自动修复，P2/P3问题标记建议
3. **混合批量诊断+修复**：同时处理细纲和正文，先细纲后正文
4. **仅诊断模式**：只输出诊断报告，不自动修改，供人工审核后决定
5. **大纲反向补充**：基于细纲诊断聚合分析，识别大纲层面缺失的设定，自动/手动补充到对应大纲模块（详见 <shared/outline-supplement.md>(../../shared/outline-supplement.md)）
6. **大纲变更影响检测**：大纲补充后自动检测变更范围，对受影响章节执行番茄平台标准评估（详见 <mode-e/outline-plot-deep-diagnosis.md>(outline-plot-deep-diagnosis.md) 能力1+2）
7. **跨章节联动修正**：章节修复后自动识别关联设定，执行级联修正确保一致性（详见 <mode-e/outline-plot-deep-diagnosis.md>(outline-plot-deep-diagnosis.md) 能力4）

## 批量诊断流程

### 总体流程

```
触发批量诊断
    │
    ├── 步骤0：范围解析与文件定位
    │   ├── 解析用户指定的章节范围（如：第1-60章）
    │   ├── 定位细纲文件/正文文件
    │   └── 确认诊断类型（细纲/正文/混合）
    │
    ├── 步骤1：逐章诊断循环
    │   └── for 每一章:
    │       ├── 1.1 读取当前章内容
    │       ├── 1.2 执行8维诊断（复用模式E诊断维度）
    │       ├── 1.3 生成单章诊断报告
    │       ├── 1.4 判断是否自动修复
    │       └── 1.5 记录诊断结果到汇总表
    │
    ├── 步骤2：自动修复执行
    │   └── for 每个P0/P1问题:
    │       ├── 2.1 定位问题位置
    │       ├── 2.2 应用修复策略
    │       ├── 2.3 修复后重新诊断验证
    │       └── 2.4 记录修复日志
    │
    ├── 步骤3：大纲反向补充（细纲诊断时自动触发）
    │   ├── 3.1 聚合所有章节的"设定不足"信号
    │   ├── 3.2 按大纲7大模块分类
    │   ├── 3.3 过滤偶发信号（<3章），保留高频信号
    │   ├── 3.4 生成缺口清单+补充方案
    │   └── 3.5 🔴缺口自动补充，🟠🟡输出建议
    │
    ├── 步骤3-A：大纲变更影响检测（大纲补充后自动触发）
    │   ├── 3-A.1 检测大纲实质性变更
    │   ├── 3-A.2 计算影响范围（直接+间接关联章节）
    │   ├── 3-A.3 对受影响章节执行番茄平台标准评估
    │   └── 3-A.4 输出变更影响报告+扩充建议
    │
    ├── 步骤3-B：跨章节联动修正（章节修复后自动触发）
    │   ├── 3-B.1 识别变动类型和关联维度
    │   ├── 3-B.2 扫描关联章节生成影响清单
    │   ├── 3-B.3 高影响项自动级联修正
    │   └── 3-B.4 修正后验证一致性
    │
    └── 步骤4：输出批量报告
        ├── 4.1 总体评分与趋势
        ├── 4.2 逐章评分明细
        ├── 4.3 问题分布统计
        ├── 4.4 修复清单
        ├── 4.5 大纲反向补充报告（新增）
        ├── 4.6 大纲变更影响报告（新增，大纲补充后自动输出）
        ├── 4.7 跨章节联动修正报告（新增，章节修复后自动输出）
        └── 4.8 改进建议
```

### 步骤0：范围解析

| 用户输入 | 解析结果 | 诊断对象 |
|---------|---------|---------|
| `细纲批量诊断 第1-60章` | 细纲范围：1-60 | 细纲文件 |
| `正文批量诊断 第1-10章` | 正文范围：1-10 | 正文文件 |
| `批量诊断 细纲第1-60章 正文第1-10章` | 细纲1-60 + 正文1-10 | 细纲+正文 |
| `细纲批量诊断 全部` | 所有已存在的细纲章节 | 细纲文件 |
| `正文批量诊断 全部` | 所有已存在的正文章节 | 正文文件 |

## 细纲批量诊断+修复

### 细纲诊断维度

对每章细纲执行以下专项诊断（区别于正文诊断）：

| 诊断维度 | 检查项 | 级别 |
|---------|--------|:--:|
| **结构完整性** | 是否包含：核心事件/阶段拆分/爽点指令/伏笔操作/感官锚点/人物状态 | P0 |
| **大纲一致性** | 核心事件是否匹配大纲6.3、爽点类型是否匹配大纲6.5、人物状态是否匹配大纲5.2 | P0 |
| **章节衔接** | 相邻章节事件是否连贯、时间线是否连续、人物状态是否平滑过渡 | P1 |
| **爽点分布** | 每章是否至少1个爽点、连续3章爽点类型是否重复、爽点强度是否递增 | P1 |
| **伏笔管理** | 伏笔编号是否唯一、伏笔埋设与回收是否匹配、是否有悬空伏笔 | P1 |
| **节奏设计** | 章节类型分布（高潮/过渡/铺垫）是否合理、连续过渡章是否超过3章 | P1 |
| **人物出场** | 新人物出场是否有铺垫、现有人物是否连续3章未出现（重要角色） | P2 |
| **字数规划** | 预估字数是否在合理范围（2000-5000字）、连续章节字数波动是否过大 | P2 |

### 细纲自动修复规则

| 问题类型 | 级别 | 自动修复策略 |
|---------|:--:|-------------|
| 缺少核心事件 | P0 | 根据前后章细纲和大纲6.3自动补全核心事件 |
| 缺少爽点指令 | P0 | 根据大纲6.5爽点类型自动生成匹配的爽点指令 |
| 缺少伏笔操作 | P0 | 检查大纲7.3伏笔规划，自动分配伏笔埋设/回收操作 |
| 大纲事件偏离 | P0 | 对照大纲6.3修正核心事件描述 |
| 人物状态跳跃 | P1 | 根据大纲5.2人物成长阶段平滑过渡 |
| 章节衔接断裂 | P1 | 在前章末尾/后章开头添加衔接过渡 |
| 爽点密度不足 | P1 | 在连续无爽点章节中插入小型爽点指令 |
| 伏笔编号冲突 | P1 | 自动重新分配唯一编号 |
| 连续过渡章过多 | P1 | 将中间过渡章升级为小型高潮章 |
| 感官锚点缺失 | P2 | 标记建议，不自动修复 |
| 字数规划偏差 | P2 | 标记建议，不自动修复 |

## 混合批量诊断（细纲+正文）

当用户同时指定细纲和正文范围时，按以下顺序执行：

```
1. 细纲批量诊断+修复（先确保细纲质量）
2. 细纲→正文映射校验（检查正文是否偏离细纲）
3. 正文批量诊断+修复（基于修复后的细纲诊断正文）
4. 输出综合报告
```

### 细纲→正文映射校验

| 字数偏差 | 正文字数与细纲预估字数偏差是否>30% | P2 |

## 仅诊断模式

当用户指定 `仅诊断不修复` 时：

1. 执行完整逐章诊断
2. 输出详细诊断报告
3. 列出所有问题和修复建议
4. **不自动修改任何文件**
5. 用户审核后可选择：
   - `全部修复`：执行所有建议修复
   - `修复P0`：仅修复P0级问题
   - `修复P0P1`：修复P0+P1级问题
   - `修复第X章`：仅修复指定章节
   - `手动修复`：用户自行修改

---

## 修复后验证

每次自动修复后，自动执行验证：

```
修复完成 → 重新诊断该章 → 对比修复前后评分 → 确认评分提升 → 记录修复日志

若修复后评分未提升或下降：
  → 回滚修复
  → 标记为"自动修复失败"
  → 转为待人工处理
```
