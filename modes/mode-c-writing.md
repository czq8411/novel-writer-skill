> 以下模板覆盖四大高价值叙事场景。当正文创作遇到对应场景类型时，加载对应模板以确保情感峰值不降级。

### 模板1：角色退场场景（POV退场）

**适用时机**：核心配角/主要角色的终章、牺牲、永别场景

```
## 退场场景写作模板

### 写作前核对
- < > 该角色的核心性格是什么？（退场行为必须符合性格）
- < > 该角色与主角的核心羁绊是什么？（退场需要回扣羁绊）
- < > 该角色退场后对主线的影响是什么？（不是「死了就完了」）

### 四拍结构
| 拍节 | 内容 | 写作要点 |
|:----:|------|---------|
| 第一拍·回顾 | 3句话以内，点明该角色与主角的核心羁绊 | 用具体物件/地点触发回忆，不写旁白 |
| 第二拍·抉择 | 角色面对牺牲/放手/守护的瞬间 | 不说「我选择」，用动作（推开门/松开手/放下剑） |
| 第三拍·台词 | 最后一句台词，必须呼应角色的核心性格或全书主题 | 控制在10字以内，越短越有力 |
| 第四拍·余味 | 一个不写情绪的微小动作（倒下的茶杯/飘落的树叶/未送出的信） | 不要写心理描写，只写客观画面 |

### 执行自检
- < > 退场场景是否让读者觉得「这是这个人应有的结局」而不是「剧情杀」？
- < > 退场后是否有至少1章的「情绪留白」让读者消化（而不是立即跳入下一场战斗）？
- < > 退场的余味镜头是否能成为全书读者讨论时的「标志性画面」？
```

### 模板2：金句呼应场景

**适用时机**：全书核心金句/主题在不同阶段的重复出现，构成呼应弧线

```
## 金句呼应写作模板

### 全书金句定义
- **核心金句**：<<一句话，必须能概括全书主题>>
- **金句的种子版本**：<<原始版本（更短/更含蓄，在第一阶段出现）>>
- **金句的完整版本**：<<完整版本（在决战/高潮阶段出现）>>

### 五段呼应弧线设计
| 呼应轮次 | 章节 | 情感阶段 | 呼应用词 | 写作手法 | 预期效果 |
|:-------:|:----:|:-------:|:--------:|:--------:|:-------:|
| 种子 | 第X章 | 故事早期 | 原始版本 | 借他人之口说出，主角不一定听懂 | 埋下种子，读者初次接触 |
| 成长 | 第X章 | 中期突破 | 变体版本 | 主角自己说出，但还不确定 | 读者意识到「这句话在变】 |
| 至暗 | 第X章 | 最低谷 | 残缺版本（话只说一半） | 主视角崩溃时低语，话没说完 | 读者担心「会不会就这样结束」 |
| 升华 | 第X章 | 最终决战 | 完整版本 | 主角在决胜时刻完整说出 | 全书情感峰值，种子终于开花 |
| 回声 | 第X章 | 尾声 | 原始版本（致敬第1次） | 由其他人/环境回声引出 | 平静的圆满感 |

### 执行自检
- < > 每一次呼应在情感积累上是否比上一次更强？（不能递减）
- < > 至暗时刻的「残缺版本」是否让读者感受到主角的脆弱？
- < > 升华时刻的完整版本是否让读者觉得「这句话值得等N章」？
```

### 模板3：身份揭示场景

**适用时机**：主角/配角发现自己与某个人物/势力的隐藏关系

```
## 身份揭示写作模板

### 六拍结构
| 节奏节拍 | 内容 | 写法要点 |
|:--------:|------|---------|
| ① 触发 | <<什么触发了揭示（物件/对话/环境）>> | 感官先行，信息后至。「他看到...他听到...然后才意识到」 |
| ② 怀疑 | <<主角开始产生怀疑>> | 用动作代替心理描写：「他的手停在半空」而不是「他心里一惊」 |
| ③ 确认 | <<揭开真相的那一刻>> | 用环境/配角反应衬托内心震撼，不写主角的心理活动 |
| ④ 对抗 | <<主角面对真相的内心斗争>> | 短句+沉默+细节，给读者留白空间 |
| ⑤ 接受 | <<做出选择>> | 不说「我接受」——用行动表示：「他抬起头」/「他站了起来」 |
| ⑥ 余味 | <<走出房间/走出世界>> | 环境暗示改变了（天亮了/下雨了/镜子裂了/鸟不叫了） |

### 执行自检
- < > 揭示的真相是否在前文有至少3处伏笔？（没有伏笔的揭示是「机械降神」）
- < > 真相是否比读者猜测的更精彩？（否则读者会失望）
- < > 揭示之后的角色选择是否让读者觉得「换做是我也会这么选」？
```

## 写作流程

### 阶段一：细纲约束加载（v21.1新增）

> **核心改进**：强化细纲到正文的约束传递，防止约束衰减

#### 1.1 强制细纲读取

**必须读取的文件**（按优先级）：
| 优先级 | 文件 | 用途 | 缺失处理 |
|:------:|------|------|---------|
| P0 | 当前章细纲 | 核心事件、爽点、钩子 | 报错，要求生成细纲 |
| P0 | 前1章细纲 | 确保连贯性 | 跳过 |
| P1 | 前3章细纲 | 伏笔追踪 | 跳过 |
| P2 | 大纲文件 | 世界观、人物设定 | 跳过 |

#### 1.2 细纲约束提取

从细纲中提取以下**强制约束项**：

```yaml
约束清单:
  核心事件:
    必须呈现: true
    允许偏离: ≤10%
    检查点: 开头/中段/结尾
  
  爽点设计:
    类型: <必须匹配细纲>
    强度: <必须≥细纲设定-1>
    位置: <必须出现在细纲指定位置>
    配角反应: <必须呈现>
  
  伏笔管理:
    必须回收: <列表>
    必须埋设: <列表>
    禁止提前回收: <列表>
  
  人物状态:
    主角状态: <必须匹配>
    关键配角: <必须匹配>
    关系变化: <如有必须呈现>
  
  章末钩子:
    类型: <悬念/危机/反转>
    强度: <必须≥细纲设定-1>
    位置: <最后300字内>
```

#### 1.3 约束验证清单（生成前）

```markdown
## 细纲约束确认

### 本章必须呈现的内容
- < > 核心事件：<细纲中的核心事件描述>
- < > 爽点类型：<类型>，强度：<X/10>
- < > 伏笔回收：<列表>
- < > 伏笔埋设：<列表>
- < > 章末钩子：<类型>，强度：<X/10>

### 人物状态基准
- 主角：<当前状态>
- 关键配角：<状态列表>

### 约束违反处理
如生成过程中发现无法满足约束：
1. 立即暂停生成
2. 标记冲突项
3. 提供解决方案（A.调整细纲 B.强制满足 C.用户决策）

#### 1.4 核心配角设定卡预装载（v22.3新增）

> 防止因遗忘角色设定导致的OOC和设定矛盾。每章正文创作前，自动装载出场角色及被提及角色的核心设定。

| 角色名 | 设定卡来源 | 本场是否出场 | 是否需装载 |
|:------|:---------|:----------:|:--------:|
| <角色> | 大纲§角色设定 / 前文记忆库 | 出场/被提及/未出场 | 出场→强制/被提及→核心3行/未出场→不装载 |

**装载规则**：
- 本章**出场**的角色：强制装载完整设定卡（性格/境界/关系/当前状态）
- 本章**被提及但未出场的角色**：可选装载核心信息（不超过3行）
- 本章**未出场的非核心角色**：不装载

**装载格式**：
```markdown
【配角设定卡】<角色名>
- 核心性格：<3个关键词>（来自大纲§角色设定）
- 当前状态：<来自前文记忆库的最新状态>
- 与本场关键关系：<与本章核心人物/事件的关系>
- ⚠️ 警惕点：<该角色在本章写作中容易OOC的陷阱>
```

### 阶段三：细纲对齐验证（v21.1新增）

> **核心改进**：生成后自动验证与细纲的一致性

#### 5.1 约束满足度检查

| 检查项 | 权重 | 通过标准 | 未通过处理 |
|--------|:---:|:-------:|-----------|
| 核心事件完整度 | 30% | ≥90% | 标记重写 |
| 爽点强度达标 | 25% | ≥细纲设定-1 | 强化爽点 |
| 伏笔正确性 | 20% | 100% | 修正或标记 |
| 钩子强度 | 15% | ≥细纲设定-1 | 强化钩子 |
| 人物一致性 | 10% | 无OOC | 修正或标记 |

#### 5.2 细纲偏离度评分

```
细纲对齐评分 = Σ(检查项得分 × 权重)

评分等级:
- 9-10分: ✅ 优秀，直接通过
- 7-8分: ⚠️ 良好，建议微调
- 5-6分: 🔶 一般，需要修改
- <5分: 🔴 不合格，必须重写
```

#### 5.3 自动修正机制

### 自动保存规则（v20.8-patch新增）

> 每章正文生成并优化完成后，必须自动保存到文件系统，无需用户手动确认。

#### 保存触发时机

| 时机 | 行为 | 用户提示 |
|------|------|---------|
| **单章生成完成后** | 自动检测路径→创建目录→保存文件 | 显示保存路径和文件大小 |

#### 保存路径规则

```
1. 获取目标路径（优先级：用户指定 > 当前工作目录 > 默认路径）
2. 检查路径权限
3. 无权限时自动降级到默认路径
4. 创建目录结构（如不存在）：正文/<卷名>/
5. 生成文件名：第XXX章_<章节标题>.md
6. 执行保存（UTF-8编码）
7. 验证保存成功（检查文件存在+大小>0）
```

#### 保存成功提示格式

**单章保存成功**：
```
✅ 第004章保存成功

   - 检查路径权限
   - 更换保存路径
   - 手动保存正文内容
```

#### 文件内容格式

保存的文件包含：
1. 章节标题（# 第X章：标题）
2. 纯净正文内容（无黏合标注、无技能注释、**无markdown分隔线**）
3. 章节结束标记（**【第X章完】**）
4. 字数统计
5. （可选）章节传递卡片（CPC）

**⚠️ 重要：正文纯净度规范（v20.8-patch2新增）**

正文内容中**严禁**出现以下元素：
- ❌ `---` 分隔线（仅允许在正文结束后、元数据前使用）
- ❌ `***` 分隔线
- ❌ `___` 分隔线
- ❌ `===` 标题线
- ❌ 任何markdown格式符号（如 `**强调**`、`> 引用`）
- ❌ HTML标签（如 `<br>`、`<div>`）

**正确的场景切换写法**：
```markdown
❌ 错误：使用分隔线
---
记忆到这里断裂了。
---

✅ 正确：使用自然过渡
记忆到这里断裂了。

沈默猛地睁开眼睛...
```

**正确的章节结构**：
```markdown
# 第006章：标题

<开头段落，直接承接上文>

<发展段落，情节推进>

<高潮段落，爽点呈现>

<结尾段落，钩子设置>

---
**【第006章完】**

**字数：约XXXX字**

---
**章节传递卡片...**
```

**注意**：`---` 只能出现在正文结束后，用于分隔正文和元数据（章节结束标记、字数统计、CPC）。正文内部禁止使用任何分隔线！
