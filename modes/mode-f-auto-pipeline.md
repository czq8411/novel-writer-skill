# 模式F：全自动创作流水线（v20.3 增强版）

> **所属**：novel-writer 核心能力库 | **触发词**：`一键创作`、`自动写`、`全自动`、`自动创作`、`恢复流水线`、`重试阶段`、`回滚阶段`
> **全局规则**：遵循主控文件第二章核心铁律

---

## 流水线概述

模式F将6大模式串联为自动化流水线，用户只需输入初始需求，AI按A→B→B-2→C→E→D顺序自动执行全流程。每阶段完成后自动进入下一阶段，仅在关键决策点暂停等待用户确认。

```
模式F触发 → 灵感构思(A) → 大纲生成(B) → 细纲规划(B-2) →
逐章创作(C)+自动校验 → 质量诊断(E) → 包装输出(D) → 完成
```

## 触发方式

输入 `一键创作`、`自动写`、`全自动` 或 `自动创作` 触发流水线。

## 流水线阶段

### 阶段0：需求收集

触发后首先收集以下信息（一次性提问，避免多轮交互）：

| 信息项 | 必填 | 说明 |
|--------|------|------|
| 小说题材 | ✅ | 玄幻/都市/言情/悬疑/末世/系统流/历史/科幻 |
| 核心创意 | ✅ | 一句话描述核心卖点（金手指/世界观/人设亮点） |
| 目标字数 | ✅ | 计划总字数或章节数 |
| 写作风格 | 可选 | 从10种风格库中选择，不选则AI根据题材自动匹配 |
| 主角设定 | 可选 | 姓名/性别/初始身份，不填则AI自动生成 |
| 特殊要求 | 可选 | 任何额外要求（如：必须有系统面板、禁止后宫等） |

### 阶段1：灵感构思（模式A）

自动调用模式A，基于用户需求生成：
- 3个创意方案（含核心梗概+金手指设计+世界观亮点）
- 用户选择1个方案（或要求融合/修改）
- 确认后进入阶段2

**暂停点**：等待用户选择创意方案。

### 阶段2：大纲生成（模式B）

自动调用模式B，基于选定方案生成：
- 7模块核心大纲（世界观/力量体系/主角成长/势力格局/人物关系/爽点密度/伏笔规划）
- 输出完整大纲供用户审阅
- 用户确认或提出修改意见

**暂停点**：等待用户确认大纲。

### 阶段3：细纲规划（模式B-2）

自动调用模式B-2，基于确认的大纲生成：
- 逐章细纲（含核心事件/阶段拆分/爽点指令/伏笔操作/感官锚点/人物状态）
- 默认首先生成前10章细纲
- 用户确认细纲格式和内容密度

**暂停点**：等待用户确认细纲格式。

### 阶段4：逐章创作（模式C）

自动进入逐章创作循环：

```
for 章节 in 细纲列表:
    1. 读取当前章细纲
    2. 读取前一章结尾（衔接用）
    3. 创作正文（遵循模式C全部规则）
    4. 自动校验（字数/禁忌词/元数据/钩子/五感/心理描写/文风）
    5. 输出正文+校验报告
    6. 更新Memory（伏笔/人物状态/剧情变动）
    7. 询问：继续下一章？/修改本章？/暂停？
```

**暂停点**：每章完成后等待用户确认（继续/修改/暂停）。

**批量模式**：用户可选择"连续写N章"，则连续创作N章后一次性展示。

**并行模式**（长篇加速）：当总章节数≥30章时，支持将章节分批派生子Agent并行创作：

```
并行创作流程：
1. 批次划分：每批5-8章，批次间不重叠，根据总章数动态调整
2. 基线同步：创作前统一读取大纲+细纲+人物设定+前文摘要，确保各Agent共享同一设定基线
3. 子Agent分配：每批分配一个子Agent，传入：
   - 当前批次的细纲列表
   - 前一批次最后一章的正文结尾（衔接用）
   - 统一的设定基线（人物/世界观/伏笔状态）
4. 并发创作：各子Agent独立创作各自批次的章节
5. 汇总校验：
   - 文风一致性校验：对比各批次语言风格/人物性格/叙事节奏
   - 衔接校验：检查批次交界处的章节衔接是否自然
   - 偏差超过阈值则统一润色
6. 主Agent汇总：合并所有章节，更新Memory，输出完整作品
```

**并行模式适用条件**：
- 总章节数≥30章
- 大纲和细纲已完整确认
- 人物设定和世界观设定已锁定
- 用户明确选择并行模式

**并行模式风险提示**：
- 文风一致性可能略有偏差（自动校验会修正）
- 适合追求效率的场景，不适合追求极致单章质量的场景

### 阶段5：质量诊断（模式E）

全部章节完成后（或用户主动触发），自动调用模式E：
- 创作数据分析仪表板（进度/字数/爽点/伏笔/人物/质量/效率）
- 7维诊断报告
- 跨卷连续性检测
- 章节节奏评分
- 输出优化建议清单

**暂停点**：展示诊断报告，等待用户决定是否修复。

### 阶段6：包装输出（模式D）

自动调用模式D，基于完整作品生成：
- 15个书名方案（5种流派各3个）
- 多平台简介适配（起点/番茄/晋江/纵横）
- 标签SEO优化方案
- 封面文案

**暂停点**：展示包装方案，等待用户选择。

### 阶段7：完成

输出创作总结：
- 总章节数/总字数
- 创作耗时统计
- 里程碑达成情况
- 下一步建议（发布平台推荐/续集规划/番外建议）

## 流水线控制（v20.3 增强版）

### 暂停与恢复

| 指令 | 行为 |
|------|------|
| `暂停` | 保存当前进度到Memory，下次输入`继续`从断点恢复 |
| `跳过XX` | 跳过当前阶段，直接进入下一阶段（如：`跳过诊断`） |
| `重做XX` | 重新执行当前阶段（如：`重做大纲`） |
| `只做XX` | 仅执行指定阶段（如：`只做大纲`→仅执行阶段2） |
| `从XX开始` | 从指定阶段开始执行（如：`从细纲开始`→跳过灵感和提纲） |

### 断点续传机制

> v20.3新增：流水线在任何阶段中断后，支持从精确断点恢复，避免重复工作。

#### 断点保存

每个阶段执行完毕后，自动将以下状态保存至Memory：

```yaml
流水线状态:
  项目名称: "[书名]"
  当前阶段: [阶段编号]
  阶段内进度:
    阶段4_逐章创作:
      已完成章节: [第X章]
      下一章节: [第Y章]
      当前批次: [批次编号]
  已完成阶段:
    - 阶段0: 需求收集 ✅
    - 阶段1: 灵感构思 ✅
    - 阶段2: 大纲生成 ✅
    - 阶段3: 细纲规划 ✅
  已生成文件:
    大纲: "[路径]"
    细纲: "[路径]"
    正文: ["[路径1]", "[路径2]", ...]
  最后保存时间: "[时间戳]"
```

#### 断点恢复流程

```
用户输入"继续"或"恢复流水线"
    │
    ├── 步骤1：读取Memory中的流水线状态
    │   ├── 状态存在且完整 → 进入步骤2
    │   ├── 状态存在但不完整 → 自动修复状态，进入步骤2
    │   └── 状态不存在 → 提示"未找到可恢复的流水线"，询问是否新建
    │
    ├── 步骤2：验证断点数据完整性
    │   ├── 检查已生成文件是否存在
    │   ├── 检查阶段间数据依赖是否满足
    │   └── 输出恢复摘要供用户确认
    │
    ├── 步骤3：用户确认后从断点恢复
    │   ├── 加载前一阶段的输出作为上下文
    │   ├── 从阶段内进度位置继续执行
    │   └── 展示："🔄 从阶段[X]恢复，已完成[Y]/[总章节]章"
    │
    └── 步骤4：继续流水线执行
```

```markdown

- 中断位置：阶段4（逐章创作），第[42]章
- 已完成：阶段0-3 ✅，阶段4 第1-[41]章 ✅
- 中断时间：[时间戳]

### 数据完整性检查
| 检查项 | 状态 |
|--------|:---:|
| 大纲文件 | ✅ 存在 |
| 细纲文件 | ✅ 存在 |
| 已生成正文(第1-41章) | ✅ 全部存在 |
| Memory状态 | ✅ 完整 |

### 恢复选项
1. **从断点继续**：从第[42]章开始创作
2. **重做当前阶段**：重新执行阶段4（保留细纲）
3. **回退到上一阶段**：回到阶段3（细纲规划）
4. **放弃并新建**：清除当前流水线，开始新项目

```

### 部分重试机制

> v20.3新增：流水线中某个阶段或章节失败时，支持精确重试，不影响已完成部分。

#### 重试粒度

| 重试粒度 | 触发方式 | 影响范围 | 适用场景 |
|---------|---------|---------|---------|
| **章节级重试** | `重试第X章` | 仅重新创作指定章节 | 单章质量不达标 |
| **批次级重试** | `重试批次[N]` | 重新创作指定批次的所有章节 | 并行模式某批次文风偏差大 |
| **阶段级重试** | `重试阶段[X]` | 重新执行指定阶段 | 大纲/细纲需要大幅调整 |
| **全量重试** | `重试全部` | 清除所有输出，从头开始 | 核心设定变更 |

#### 章节级重试流程

```
用户输入"重试第X章"
    │
    ├── 步骤1：定位目标章节
    │   ├── 检查章节是否存在于已完成列表
    │   └── 不存在 → 提示"第X章尚未创作"
    │
    ├── 步骤2：分析重试原因（询问用户或自动检测）
    │   ├── 用户指定原因 → 记录到重试日志
    │   └── 自动检测 → 分析上一版的问题（字数/质量/风格）
    │
    ├── 步骤3：备份原章节
    │   └── 将原章节移至版本历史（参见mode-c/version-management.md）
    │
    ├── 步骤4：重新创作
    │   ├── 读取前一章结尾（衔接用）
    │   ├── 读取当前章细纲
    │   ├── 应用重试原因中的改进方向
    │   └── 创作新版本
    │
    ├── 步骤5：级联影响检测
    │   ├── 检测本章变更是否影响后续章节
    │   ├── 人物状态变化 → 标记后续章节需复核
    │   ├── 伏笔变更 → 更新伏笔追踪表
    │   └── 剧情走向变化 → 警告用户评估后续章节
    │
    └── 步骤6：输出新章节+影响报告
```

#### 重试日志

```markdown

| 时间 | 章节 | 重试原因 | 原评分 | 新评分 | 级联影响 |
|------|:---:|------|:---:|:---:|------|

```

### 异常回滚机制

> v20.3新增：流水线执行过程中遇到不可恢复的错误时，自动回滚到上一个稳定状态，保护已完成的工作。

#### 异常分类与处理策略

| 异常类型 | 严重度 | 自动处理 | 回滚范围 |
|---------|:---:|:---:|------|
| **文件写入失败** | P1 | ✅ 自动重试3次 | 当前章节 |
| **Memory更新失败** | P1 | ✅ 自动重试+降级保存 | 当前章节 |
| **生成内容质量不达标** | P2 | ⚠️ 自动标记+通知用户 | 当前章节 |
| **设定冲突检测** | P2 | ⚠️ 暂停+展示冲突+等待决策 | 当前阶段 |
| **依赖文件丢失** | P0 | 🔴 立即暂停+回滚 | 回滚到上一个完整阶段 |
| **连续3次重试失败** | P0 | 🔴 暂停流水线+建议手动介入 | 回滚到上一个稳定状态 |

#### 回滚流程

```
异常触发
    │
    ├── 步骤1：异常识别与分级
    │   ├── P0级（致命）：立即暂停流水线
    │   ├── P1级（严重）：自动重试，3次失败后升级为P0
    │   └── P2级（警告）：记录日志，继续执行
    │
    ├── 步骤2：确定回滚点
    │   ├── 计算上一个稳定状态（最近一次全部成功的阶段完成点）
    │   ├── P0异常 → 回滚到上一个完整阶段
    │   └── P1异常3次失败 → 回滚到当前阶段开始点
    │
    ├── 步骤3：执行回滚
    │   ├── 保留回滚点之前的所有输出文件
    │   ├── 清除回滚点之后的不完整输出
    │   ├── 恢复Memory到回滚点状态
    │   └── 记录回滚日志
    │
    ├── 步骤4：输出回滚报告
    │   └── 展示异常原因、回滚范围、受影响文件、恢复建议
    │
    └── 步骤5：等待用户决策
        ├── 从回滚点继续
        ├── 手动修复后继续
        └── 放弃流水线
```

### 流水线健康监控

> v20.3新增：实时监控流水线执行状态，主动发现潜在问题。

| 监控项 | 检测方式 | 预警条件 | 处理 |
|--------|---------|---------|------|
| 连续重试次数 | 重试日志计数 | 同一章节重试≥3次 | 暂停并建议检查细纲/大纲 |
| 质量趋势下降 | 章节评分趋势 | 连续3章评分下降 | 暂停并建议诊断 |
| 字数异常波动 | 章节字数统计 | 单章字数偏离均值>40% | 标记并建议复核 |
| 阶段耗时异常 | 阶段执行时间 | 单阶段耗时>预期2倍 | 记录日志，继续执行 |
| Memory增长异常 | Memory文件大小 | 单次更新>10KB | 建议压缩或归档旧状态 |

## 与手动模式的关系

- 模式F是6大模式的自动化编排，不替代任何模式
- 用户可随时退出流水线，切换到手动模式（如：`退出流水线，手动写第5章`）
- 流水线中每个阶段的质量标准与手动模式完全一致
- 流水线中的暂停点确保用户对关键决策保持控制权

### 协作技能与接口

#### 1. chinese-novelist 协作

| 协作场景 | 触发方式 | 数据传递 | 返回处理 |
|---------|---------|---------|---------|
| **灵感碰撞** | `与chinese-novelist碰撞灵感 [主题]` | 传递当前大纲/创意 | 融合双方灵感，输出增强方案 |
| **风格参考** | `参考chinese-novelist风格写第X章` | 传递章节细纲+风格要求 | 按参考风格创作，保持novel-writer质量体系 |
| **交叉诊断** | `让chinese-novelist诊断第X章` | 传递章节正文 | 获取外部诊断意见，与模式E诊断合并 |

**协作流程**：
```
1. novel-writer发起协作请求 → 传递上下文数据包
2. 协作技能处理请求 → 返回处理结果
3. novel-writer接收结果 → 融合到当前创作流程
4. 更新Memory记录协作历史
```

**数据包格式**：
```json
{
  "collaborationRequest": {
    "from": "novel-writer",
    "to": "chinese-novelist",
    "type": "灵感碰撞",
    "context": {
      "currentProject": "《苍穹之上》",
      "currentChapter": 42,
      "outline": "（大纲摘要）",
      "requestDetail": "需要为第43章设计一个出人意料的秘境来客身份"
    },
    "expectedOutput": "3个身份方案，每个含揭示方式和后续影响"
  }
}
```

### 协作触发词

| 触发词 | 协作技能 | 说明 |
|--------|---------|------|
| `碰撞灵感` `交叉诊断` `风格参考` | chinese-novelist | 与chinese-novelist技能协作 |
| `设计封面` `世界地图` `人物立绘` | canvas-design | 与canvas-design技能协作 |
| `作品提案` `创作计划PPT` | pptx | 与pptx技能协作 |
| `导出Word` `创作报告` | docx | 与docx技能协作 |

### 协作原则

1. **主从关系**：novel-writer为主技能，协作技能为辅助，novel-writer负责最终决策
2. **数据隔离**：协作过程中传递的数据不污染novel-writer的Memory和创作状态
3. **结果融合**：协作结果需经novel-writer审核和融合，不直接覆盖现有内容
4. **可追溯**：所有协作记录保存在Memory的collaborationHistory中
