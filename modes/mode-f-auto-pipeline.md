# 模式F：全自动创作流水线

> **所属**：novel-writer 核心能力库 | **触发词**：`一键创作`、`自动写`、`全自动`、`自动创作`
> **全局规则**：遵循主控文件第二章核心铁律

---

## 流水线概述

模式F将6大模式串联为自动化流水线，用户只需输入初始需求，AI按A→B→B-2→C→E→D顺序自动执行全流程。每阶段完成后自动进入下一阶段，仅在关键决策点暂停等待用户确认。

```
模式F触发 → 灵感构思(A) → 大纲生成(B) → 细纲规划(B-2) →
逐章创作(C)+自动校验 → 质量诊断(E) → 包装输出(D) → 完成
```

## 触发方式

输入 `一键创作`、`自动写`、`全自动` 或 `自动创作` 触发流水线。

## 流水线阶段

### 阶段0：需求收集

触发后首先收集以下信息（一次性提问，避免多轮交互）：

| 信息项 | 必填 | 说明 |
|--------|------|------|
| 小说题材 | ✅ | 玄幻/都市/言情/悬疑/末世/系统流/历史/科幻 |
| 核心创意 | ✅ | 一句话描述核心卖点（金手指/世界观/人设亮点） |
| 目标字数 | ✅ | 计划总字数或章节数 |
| 写作风格 | 可选 | 从10种风格库中选择，不选则AI根据题材自动匹配 |
| 主角设定 | 可选 | 姓名/性别/初始身份，不填则AI自动生成 |
| 特殊要求 | 可选 | 任何额外要求（如：必须有系统面板、禁止后宫等） |

### 阶段1：灵感构思（模式A）

自动调用模式A，基于用户需求生成：
- 3个创意方案（含核心梗概+金手指设计+世界观亮点）
- 用户选择1个方案（或要求融合/修改）
- 确认后进入阶段2

**暂停点**：等待用户选择创意方案。

### 阶段2：大纲生成（模式B）

自动调用模式B，基于选定方案生成：
- 7模块核心大纲（世界观/力量体系/主角成长/势力格局/人物关系/爽点密度/伏笔规划）
- 输出完整大纲供用户审阅
- 用户确认或提出修改意见

**暂停点**：等待用户确认大纲。

### 阶段3：细纲规划（模式B-2）

自动调用模式B-2，基于确认的大纲生成：
- 逐章细纲（含核心事件/阶段拆分/爽点指令/伏笔操作/感官锚点/人物状态）
- 默认首先生成前10章细纲
- 用户确认细纲格式和内容密度

**暂停点**：等待用户确认细纲格式。

### 阶段4：逐章创作（模式C）

自动进入逐章创作循环：

```
for 章节 in 细纲列表:
    1. 读取当前章细纲
    2. 读取前一章结尾（衔接用）
    3. 创作正文（遵循模式C全部规则）
    4. 自动校验（字数/禁忌词/元数据/钩子/五感/心理描写/文风）
    5. 输出正文+校验报告
    6. 更新Memory（伏笔/人物状态/剧情变动）
    7. 询问：继续下一章？/修改本章？/暂停？
```

**暂停点**：每章完成后等待用户确认（继续/修改/暂停）。

**批量模式**：用户可选择"连续写N章"，则连续创作N章后一次性展示。

**并行模式**（长篇加速）：当总章节数≥30章时，支持将章节分批派生子Agent并行创作：

```
并行创作流程：
1. 批次划分：每批5-8章，批次间不重叠，根据总章数动态调整
2. 基线同步：创作前统一读取大纲+细纲+人物设定+前文摘要，确保各Agent共享同一设定基线
3. 子Agent分配：每批分配一个子Agent，传入：
   - 当前批次的细纲列表
   - 前一批次最后一章的正文结尾（衔接用）
   - 统一的设定基线（人物/世界观/伏笔状态）
4. 并发创作：各子Agent独立创作各自批次的章节
5. 汇总校验：
   - 文风一致性校验：对比各批次语言风格/人物性格/叙事节奏
   - 衔接校验：检查批次交界处的章节衔接是否自然
   - 偏差超过阈值则统一润色
6. 主Agent汇总：合并所有章节，更新Memory，输出完整作品
```

**并行模式适用条件**：
- 总章节数≥30章
- 大纲和细纲已完整确认
- 人物设定和世界观设定已锁定
- 用户明确选择并行模式

**并行模式风险提示**：
- 文风一致性可能略有偏差（自动校验会修正）
- 适合追求效率的场景，不适合追求极致单章质量的场景

### 阶段5：质量诊断（模式E）

全部章节完成后（或用户主动触发），自动调用模式E：
- 创作数据分析仪表板（进度/字数/爽点/伏笔/人物/质量/效率）
- 7维诊断报告
- 跨卷连续性检测
- 章节节奏评分
- 输出优化建议清单

**暂停点**：展示诊断报告，等待用户决定是否修复。

### 阶段6：包装输出（模式D）

自动调用模式D，基于完整作品生成：
- 15个书名方案（5种流派各3个）
- 多平台简介适配（起点/番茄/晋江/纵横）
- 标签SEO优化方案
- 封面文案

**暂停点**：展示包装方案，等待用户选择。

### 阶段7：完成

输出创作总结：
- 总章节数/总字数
- 创作耗时统计
- 里程碑达成情况
- 下一步建议（发布平台推荐/续集规划/番外建议）

## 流水线控制

### 暂停与恢复

| 指令 | 行为 |
|------|------|
| `暂停` | 保存当前进度到Memory，下次输入`继续`从断点恢复 |
| `跳过XX` | 跳过当前阶段，直接进入下一阶段（如：`跳过诊断`） |
| `重做XX` | 重新执行当前阶段（如：`重做大纲`） |
| `只做XX` | 仅执行指定阶段（如：`只做大纲`→仅执行阶段2） |
| `从XX开始` | 从指定阶段开始执行（如：`从细纲开始`→跳过灵感和提纲） |

### 进度展示

每个阶段完成后展示进度条：

```
[████████░░░░░░░░░░░░] 40%  阶段3/7：细纲规划 ✅
[████████████░░░░░░░░] 60%  阶段4/7：逐章创作 ⏳
```

## 与手动模式的关系

- 模式F是6大模式的自动化编排，不替代任何模式
- 用户可随时退出流水线，切换到手动模式（如：`退出流水线，手动写第5章`）
- 流水线中每个阶段的质量标准与手动模式完全一致
- 流水线中的暂停点确保用户对关键决策保持控制权

## 跨技能协作接口

> novel-writer支持与其他Trae技能协作，实现创作流程的扩展和增强。

### 协作架构

```
                    ┌─────────────────┐
                    │   novel-writer   │
                    │   （核心创作）    │
                    └───┬───────┬─────┘
                        │       │
          ┌─────────────┼───────┼─────────────┐
          │             │       │             │
    ┌─────▼─────┐ ┌────▼──┐ ┌──▼──────┐ ┌────▼─────┐
    │ chinese-  │ │canvas-│ │  pptx   │ │  docx    │
    │ novelist  │ │design │ │         │ │          │
    └───────────┘ └───────┘ └─────────┘ └──────────┘
```

### 协作技能与接口

#### 1. chinese-novelist 协作

| 协作场景 | 触发方式 | 数据传递 | 返回处理 |
|---------|---------|---------|---------|
| **灵感碰撞** | `与chinese-novelist碰撞灵感 [主题]` | 传递当前大纲/创意 | 融合双方灵感，输出增强方案 |
| **风格参考** | `参考chinese-novelist风格写第X章` | 传递章节细纲+风格要求 | 按参考风格创作，保持novel-writer质量体系 |
| **交叉诊断** | `让chinese-novelist诊断第X章` | 传递章节正文 | 获取外部诊断意见，与模式E诊断合并 |

**协作流程**：
```
1. novel-writer发起协作请求 → 传递上下文数据包
2. 协作技能处理请求 → 返回处理结果
3. novel-writer接收结果 → 融合到当前创作流程
4. 更新Memory记录协作历史
```

**数据包格式**：
```json
{
  "collaborationRequest": {
    "from": "novel-writer",
    "to": "chinese-novelist",
    "type": "灵感碰撞",
    "context": {
      "currentProject": "《苍穹之上》",
      "currentChapter": 42,
      "outline": "（大纲摘要）",
      "requestDetail": "需要为第43章设计一个出人意料的秘境来客身份"
    },
    "expectedOutput": "3个身份方案，每个含揭示方式和后续影响"
  }
}
```

#### 2. canvas-design 协作

| 协作场景 | 触发方式 | 数据传递 | 返回处理 |
|---------|---------|---------|---------|
| **封面设计** | `为《XXX》设计封面` | 传递书名+简介+风格要求 | 输出封面图片 |
| **世界地图** | `生成世界地图` | 传递世界观设定+场景列表 | 输出地图图片 |
| **人物立绘** | `生成[角色名]立绘` | 传递人物设定+外貌描写 | 输出人物立绘 |

#### 3. pptx 协作

| 协作场景 | 触发方式 | 数据传递 | 返回处理 |
|---------|---------|---------|---------|
| **作品提案** | `生成作品提案PPT` | 传递大纲+市场分析 | 输出PPT文件 |
| **创作计划** | `生成创作计划PPT` | 传递章节规划+里程碑 | 输出PPT文件 |

#### 4. docx 协作

| 协作场景 | 触发方式 | 数据传递 | 返回处理 |
|---------|---------|---------|---------|
| **导出作品** | `导出为Word文档` | 传递全部章节正文 | 输出格式化的.docx文件 |
| **创作报告** | `生成创作报告` | 传递诊断数据+统计 | 输出创作报告.docx |

### 协作触发词

| 触发词 | 协作技能 | 说明 |
|--------|---------|------|
| `碰撞灵感` `交叉诊断` `风格参考` | chinese-novelist | 与chinese-novelist技能协作 |
| `设计封面` `世界地图` `人物立绘` | canvas-design | 与canvas-design技能协作 |
| `作品提案` `创作计划PPT` | pptx | 与pptx技能协作 |
| `导出Word` `创作报告` | docx | 与docx技能协作 |

### 协作原则

1. **主从关系**：novel-writer为主技能，协作技能为辅助，novel-writer负责最终决策
2. **数据隔离**：协作过程中传递的数据不污染novel-writer的Memory和创作状态
3. **结果融合**：协作结果需经novel-writer审核和融合，不直接覆盖现有内容
4. **可追溯**：所有协作记录保存在Memory的collaborationHistory中
