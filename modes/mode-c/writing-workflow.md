# 模式C子模块：写作流程与项目管理

> 按需加载：首次进入模式C或切换写作模式时加载本文件。
> 全局规则：遵循主控文件第二章核心铁律

---

## 一、写作模式

| 模式 | 参数值 | 适用场景 | 说明 |
|------|--------|---------|------|
| **串行**（默认） | `serial` | 单章精写 | 逐章创作，质量优先 |
| **批量串行** | `batch` | 连续多章 | 一次性生成N章细纲后逐章创作正文 |
| **并行** | `parallel` | 多章同时产出 | 将连续N章分配给子Agent并行创作，通过`writing-plan.json`协调状态 |
| **连续创作** | `continuous` | 快速产出完整作品 | 确认起始章节后自动逐章创作直到完成，仅在发现问题时暂停 |

**并行模式额外要求**：
- 创作前统一读取大纲+细纲+人物设定，确保各Agent共享同一设定基线
- 创作后执行文风一致性校验：对比各章语言风格、人物性格、叙事节奏，偏差超过阈值则统一润色

### 连续创作模式详解

用户确认起始章节和终止章节（或"写到完"）后，AI自动逐章创作，全程无需逐章确认。

```
连续创作流程：
1. 用户确认：起始章节 + 终止章节（或"写到完"）
2. AI逐章创作，每章完成后静默执行7维自动校验
3. 校验通过 → 自动进入下一章，输出简短进度提示
4. 校验不通过 → 自动修复（最多2轮），仍不通过则暂停询问用户
5. 每5章输出一次进度摘要（含字数统计、爽点密度、伏笔状态）
6. 全部完成 → 输出创作总结
```

**连续创作进度提示格式**：
```
✅ 第042章完成（3850字）→ 自动进入第043章...
✅ 第043章完成（4100字）→ 自动进入第044章...
⚠️ 第044章字数不足（2800字），自动扩充中...
✅ 第044章修复完成（3650字）→ 自动进入第045章...

📊 进度摘要（第041-045章）
| 章节 | 字数 | 爽点 | 伏笔 | 校验 |
|------|------|------|------|------|
| 041 | 4200 | ✅ | F12回收 | ✅ |
| 042 | 3850 | ✅ | — | ✅ |
| 043 | 4100 | ✅ | F15埋下 | ✅ |
| 044 | 3650 | ✅ | — | ⚠️修复后通过 |
| 045 | 3900 | ✅ | — | ✅ |
```

**连续创作暂停条件**（仅在以下情况暂停询问用户）：
- 自动修复2轮后校验仍不通过
- 检测到与大纲/细纲的重大偏离
- 检测到人物OOC（性格行为严重不符）
- 用户主动输入"暂停"

## 二、章节自动编号与项目管理

> 首次创作时自动创建项目目录结构，后续章节自动编号，无需用户手动管理。

### 项目初始化

首次进入模式C时，自动检测并创建项目结构：

```
检测流程：
1. 执行 pwd 获取当前工作目录
2. 检查是否存在 ./正文/ 目录
3. 若不存在 → 创建完整项目结构（正文/、细纲/、大纲/均在 pwd 下）
4. 若存在 → 检测已有章节，自动分配下一章编号
```

**自动创建的目录结构**（所有目录在当前工作目录下）：
```
./
├── 记忆库.md
├── 大纲/
│   └── [书名]_核心大纲.md
├── 细纲/
│   └── 第一卷_XXX/
│       ├── 第001章_[标题].md
│       └── 细纲目录.md
└── 正文/
    └── 第一卷_XXX/
        └── 第001章_[标题].md
```

### 章节自动编号

| 操作 | 行为 |
|------|------|
| 首次创作 | 自动编号为"第001章" |
| 续写新章 | 扫描已有章节文件，自动分配下一章编号 |
| 插入章节 | 用户指定编号，后续章节自动顺延 |
| 章节重排 | 输入 `重排章节` 触发全局编号重排 |

**编号格式**：`第XXX章_[章节标题].md`（XXX为3位数字，不足补零）

## 三、输入格式

| 输入项 | 来源 | 使用方式 | 提取要点 |
|--------|------|---------|---------|
| 当前章细纲 | Filesystem读取 | **核心遵循** | 核心事件、阶段拆分、爽点指令、伏笔操作、感官锚点、人物状态 |
| 前一章正文 | Filesystem读取 | **仅读结尾500字** | 场景终点、人物状态、未解决钩子、确保衔接自然 |

## 四、前置要求（撰写前必读）

> **核心原则**：大纲信息已通过细纲间接引用，写正文时**严格遵循当前章细纲**即可，无需直接查阅大纲。

### 第一层：细纲执行（核心聚焦）
细纲已包含大纲所有关键信息，写作时严格遵循以下指令：

| 细纲要素 | 执行要求 | 检查点 |
|---------|---------|--------|
| **核心事件** | 必须成为正文主线骨架 | 是否偏离细纲一句话剧情？ |
| **阶段拆分** | 3-5个阶段对应正文段落结构 | 字数分配是否符合细纲？ |
| **爽点指令** | 每个爽点必须逐一实现，不得遗漏或淡化 | 四段式结构是否完整？ |
| **伏笔操作** | 回收（FXX）和埋下（FXX）必须精确执行 | 伏笔编号是否正确？ |
| **人物状态** | 表格中每个人物的情绪/状态必须准确呈现 | 是否符合细纲表格？ |
| **感官锚点** | 每个场景的感官锚点必须在正文中体现 | 是否调用非视觉感官？ |
| **大纲引用** | 细纲中标注的大纲模块编号已足够 | 仅在细纲与大纲冲突时回查 |

### 第二层：前文衔接（精简聚焦）
仅读取**前一章正文结尾500字**，确认以下衔接点：

| 衔接要素 | 确认内容 | 处理方式 |
|---------|---------|---------|
| **场景终点** | 前一章结尾场景是什么？ | 本章开头场景必须能自然承接 |
| **人物状态** | 主角境界、装备、情绪如何？ | 本章开头状态必须一致 |
| **未解决钩子** | 前一章留下了什么悬念？ | 本章开头必须回应或延续 |
| **文风延续** | 前一章的叙述节奏、语言风格 | 保持文风统一 |

---

## 五、排版与视觉要求

- **移动端适配**：每段不超过3行（手机屏幕宽度），段落间留空行
- **对话独立**：对话必须独立成段，严禁大段对话堆砌
- **视觉留白**：关键剧情转折处，使用单句成段，制造节奏感

---

## 六、写作规则

**通用规则**：严格遵循主控文件第二章核心铁律（2.3去AI味、2.4沉浸式体验、2.5数据导向），包括禁忌词汇黑名单、质感描写词库、五感调用、潜台词设计、动作代替标签、语言纯正性等全部约束。

**模式特有规则**：

### 元数据隔离规则
大纲/细纲中的人物设定标签、伏笔编号（如V1-014、F01）、元数据标记（如#记忆更新）、细纲指令（如【爽点】）是**作者参考信息**，严禁直接输出到正文。

**过滤清单**（输出前扫描删除）：
- 人物属性标签格式（"姓名。身份。血脉。编号。"）
- 伏笔编号（V1-XXX、FXX、"伏笔：XXX"）
- 元数据标记（#记忆更新、#伏笔回收）
- 细纲指令标记（【爽点】、【钩子】）
- 作者备注（（此处需加强）、（待补充））

**转化示例**：
- ❌ "李铁牛。猎户之子。远古战族血脉。V1-014。"
- ✅ "那少年叫李铁牛，猎户的儿子。林凡的感知扫过他时，察觉到那具普通身躯里沉睡着某种古老而狂暴的力量。"
- ❌ "埋下伏笔V1-014"
- ✅ "李铁牛低头看着掌心那道暗红色的纹路，像一条沉睡的蛇。"

### 心理描写限制
- **限制长度**：单次心理独白严禁超过3行
- **穿插动作**：心理描写必须穿插在动作或对话中，严禁大段静止思考

### 节奏与爽点
遵循主控文件2.5数据导向原则，对照细纲中的爽点指令执行。

### 细纲逐项核对机制
写作中必须实时对照细纲执行，确保核心事件、爽点类型、人物状态、伏笔、场景、境界、时间线与细纲完全对齐。核对流程：写作前读取细纲 → 写作中逐项对照 → 完成后自检。
