### 优先级分级

| 级别 | 标识 | 调用时机 | 量化影响 |
|:---:|:---:|---------|:---:|
| P0 | 🔴 强制 | 必须调用，存在致命问题（战力崩坏、伏笔未回收） | -20分 |
| P1 | 🟠 高优 | 应该调用，关联度高（章节衔接不畅、爽点不足） | -10分 |
| P2 | 🟡 标准 | 建议调用，提升完整性 | -5分 |
| P3 | 🟢 可选 | 可选调用，性能优先时跳过 | -2分 |

### 执行流程

1. **输入解析** → 判定诊断意图  
2. **子技能识别** → 匹配核心/辅助子技能  
3. **优先级排序** → 计算优先级并排序  
4. **依赖分析** → 确定加载顺序  
5. **并行/顺序执行** → 独立技能并行，依赖技能按序  
6. **结果聚合** → 生成综合报告  
7. **优化触发** → 分析结果触发优化功能

## 自动化诊断修复模式

**触发词**：`自动修复`、`一键修复`、`自动优化`、`修复循环`

### 循环流程

```
诊断 → 分析问题 → 检查停止条件 → 自动修复 → 循环/停止
```

### 停止条件

| 条件 | 默认值 |
|-----|:---:|
| 连续2轮无P0/P1问题 | ✅ |
| 最大修复轮数 | 5轮 |
| 综合评分≥8分 | ✅ |
| 连续2轮评分无提升 | ✅ |

### 修复优先级

1. 🔴 P0级（致命问题）→ 立即修复  
2. 🟠 P1级（重要问题）→ 优先修复  
3. 🟡 P2级（一般问题）→ 按需修复  
4. 🟢 P3级（细节问题）→ 跳过/最后处理

### 控制命令

| 命令 | 功能 |
|------|------|
| `暂停` | 保存进度，可继续 |
| `继续` | 恢复暂停的循环 |
| `终止` | 保存进度，退出 |
| `取消` | 取消当前操作 |

### 异常处理

| 失败类型 | 处理方式 |
|---------|---------|
| 单问题修复失败 | 跳过，记录原因 |
| 连续失败≥3次 | 暂停等待用户确认 |
| 诊断失败 | 重试最多3次 |
| 系统异常 | 自动保存，安全退出 |

## 自动迭代修复闭环（v22.1新增·模式E→模式B联动）

> **核心改进**：诊断结果不再「看过即止」，而是自动生成修复方案，引导进入模式B重生成优化版大纲。

### 闭环流程

```
用户完成模式B大纲生成
        │
        ▼
┌──────────────────────────────────┐
│ 步骤1：自动进入模式E大纲诊断       │
│ - 加载20维诊断                   │
│ - 生成评分+问题清单+优先级排序     │
└──────────────┬───────────────────┘
               │
               ▼
┌──────────────────────────────────┐
│ 步骤2：生成修复方案               │
│ - 每个P0/P1问题对应一个修复模板   │
│ - 模板包含：问题描述+建议修改值    │
│ - 修复方案聚合为「重生成指令」     │
└──────────────┬───────────────────┘
               │
               ▼
┌──────────────────────────────────┐
│ 步骤3：展示给用户                │
│ - 诊断评分                       │
│ - 问题清单（分级）               │
│ - 修复方案预览                   │
│ - 用户选择：                     │
│   A. 自动重生成（一键执行）       │
│   B. 手动调整部分字段            │
│   C. 暂不接受，手动修改          │
└──────────────┬───────────────────┘
               │ 用户选择A或B
               ▼
┌──────────────────────────────────┐
│ 步骤4：自动进入模式B重生成         │
│ - 加载原始大纲基准                 │
│ - 应用修复方案中的字段修改         │
│ - 调用模式B生成优化版大纲          │
│ - 生成后再次进入模式E验证          │
└──────────────┬───────────────────┘
               │
               ▼
┌──────────────────────────────────┐
│ 步骤5：循环终止条件               │
│ - 评分≥8分 → ✅ 通过，进入模式B-2 │
│ - 连续2轮评分无提升 → 暂停等待   │
│ - 最大循环数5轮 → 强制退出        │
└──────────────────────────────────┘
```

## 上下文控制规则（v22.2）

### 诊断子技能加载
- 用户输入诊断触发词后，仅加载「智能路由引擎」和匹配的「核心子技能+辅助子技能」
- 未激活的诊断子技能不进入生成上下文
- 15个 mode-e/* 子模块按场景按需加载，不一次性注入

### 子技能加载时机

| 用户输入 | 加载的内容 | 不加载的内容 |
|---------|-----------|-------------|
| `诊断 第X章` | auto-validation + style-fingerprint + reader-retention | 大纲诊断/细纲诊断子模块 |
| `大纲诊断` | outline-evaluation + setting-detection + outline-plot-deep-diagnosis | 正文诊断/细纲诊断子模块 |
| `检测AI味` | ai-flavor-detection + style-fingerprint | 跨卷/仪表板/修复循环 |

### 优化目标
- 单次诊断：加载量控制在 **400-500 行**（编排器 + 2-3个子技能）
- 避免一次性加载全部15个子模块（约6,000行），只加载当前诊断类型需要的

## 优先级排序机制

**执行原则**：按优先级顺序执行，P0级问题立即阻断。

1. **第一步**：排查P0级致命问题  
2. **第二步**：分析P1级重要问题  
3. **第三步**：检查P2级一般问题  
4. **第四步**：处理P3级细节问题  
5. **第五步**：验证前后章衔接流畅度  
6. **第六步**：验证大纲一致性  
7. **第七步**：执行剩余诊断维度  
8. **第八步**：计算综合评分并定级  
9. **第九步**：触发优化功能

### 优化功能触发规则

| 诊断条件 | 触发优化功能 |
|---------|-------------|
| 人物群像深度<6分 | 情感光谱系统 |
| 爽点质量<6分 | 情感光谱系统 |
| 写作质量<6分 | 通感写作技法 |
| 世界观展开节奏<6分 | 通感写作技法 |
| 大纲一致性<6分 | 长篇创作支持 |
| 剧情结构<6分 | 长篇创作支持 |
| 当前章节≥50章 | 长篇创作支持 |

## 全书回眸诊断（v22.3新增）

> 完成30章+长段落后的回溯性诊断。人类大神作家会反复修改前文来呼应后文，AI不会自觉回溯。本模式在完成长段落后反推前文，提出「可以增强前后呼应的修改建议」。

### 触发条件

| 触发方式 | 条件 | 产出 |
|---------|------|------|
| 自动触发 | 模式C完成第30/50/100章后 | 全书回眸诊断报告 |
| 手动触发 | 用户输入「回眸诊断」 | 同上 |

### 诊断维度

| 维度 | 检测内容 | 示例发现 |
|:----|---------|---------|
| **首尾呼应潜力** | 检测第1章与最新章的元素关联性 | 「第1章出现的表壳划痕在第30章未被呼应——建议在第30章增加主角注意到划痕的细节，形成闭环」 |
| **伏笔偏离检测** | 对比初始伏笔定义与最新章实际状态 | 「F05（爷爷的铁盒）原始设定内含7块表，正文第2章写'盒子里有表'但未明确数量」 |
| **角色成长弧线完整性** | 检测主角从起点到终点的成长是否连续 | 「第1章主角最在意的是一只表的状况，第30章但他关心的重点变成了人——这个转变过程是否铺垫充分？」 |
| **金句呼应收割** | 检测前文金句在后文是否被呼应 | 「第3章出场的金句'你不想走吗'未被第30章呼应——建议在结局增加主角对老钟说出同样台词的场景」 |

### 金句终局回声调度（v22.3新增）

> 全书完成50章+后，自动检测终局是否调用了前文的金句形成情感回声。避免写出「很好的终局，但本来可以更好」的遗憾。

**触发条件**：全书完成第50/100章后，或用户输入「金句回声调度」。

**检测规则**：
1. 从金句归档系统（preferences.md）中提取全书出现频率最高的**前3个金句**
2. 检测终局章节（最后3章）中是否引用了至少1个金句（直接引用或同一句式的变体）
3. 若未引用 → 生成「建议在终局中增加金句回响」的建议
4. 若已引用 → 检查引用位置是否在终局的**情感峰值位置**（最后30%篇幅内），若不在则建议调整

**输出示例**：

| 「修的不是表，是时间」 | 第15章 | ✅ 已引用（第49章） | ✅ 位置合理 |
