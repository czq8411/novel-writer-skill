# 共享模块：用户反馈收集机制

> 按需加载：当用户主动提供反馈、报告问题或建议改进时。
> 全局规则：遵循主控文件第二章核心铁律

---

## 一、反馈收集概述

本模块定义用户反馈的收集、处理和迭代机制，确保novel-writer-skill持续优化。

**反馈类型**：
- **功能建议**：新增功能、改进现有功能
- **问题报告**：Bug、错误、异常行为
- **使用体验**：易用性、流程优化
- **内容质量**：生成内容的质量反馈
- **其他**：任何有助于改进的反馈

**收集渠道**：
- 会话内直接反馈
- 文件内注释反馈
- 结构化反馈表单

## 二、反馈收集流程

### 2.1 主动收集

当完成一次创作任务后，系统可主动询问：

```
创作任务已完成！为了更好地为您服务，请花1分钟提供反馈：

1. 本次创作是否符合您的预期？
   [1] 完全符合  [2] 基本符合  [3] 一般  [4] 不太符合  [5] 完全不符合

2. 您对以下方面的满意度（1-5分）：
   - 内容质量：____
   - 创作效率：____
   - 交互体验：____

3. 您遇到的问题或建议（可选）：
   ____________________

感谢您的反馈！这将帮助我们持续改进。
```

### 2.2 被动收集

用户可随时输入以下指令提供反馈：

| 指令 | 功能 |
|------|------|
| `反馈` / `feedback` | 进入反馈模式 |
| `建议` / `suggest` | 提交功能建议 |
| `报告问题` / `bug` | 报告Bug或错误 |
| `评分` / `rate` | 对当前服务评分 |

## 三、反馈分类与处理

### 3.1 反馈分类标准

| 分类 | 定义 | 优先级 | 响应时间 |
|------|------|--------|---------|
| **P0-紧急** | 系统错误、数据丢失、安全问题 | 最高 | 24小时内 |
| **P1-高** | 核心功能缺陷、严重影响体验 | 高 | 3天内 |
| **P2-中** | 一般功能问题、改进建议 | 中 | 1周内 |
| **P3-低** | 优化建议、细节调整 | 低 | 2周内 |
| **P4-待评估** | 需要进一步讨论的需求 | 待定 | 不确定 |

### 3.2 反馈处理流程

```
用户提交反馈
    ↓
自动分类（基于关键词）
    ↓
人工审核（确认分类和优先级）
    ↓
├─→ P0/P1：立即处理，紧急修复
├─→ P2：纳入迭代计划
├─→ P3：记录待后续版本考虑
└─→ P4：讨论后决定
    ↓
处理后通知用户
    ↓
记录到改进日志
```

## 五、反馈数据分析

### 5.1 数据收集维度

| 维度 | 说明 | 用途 |
|------|------|------|
| **反馈类型分布** | 各类反馈的占比 | 识别主要改进方向 |
| **功能模块分布** | 反馈涉及的功能模块 | 定位问题集中区域 |
| **时间趋势** | 反馈数量随时间变化 | 评估改进效果 |
| **用户满意度** | 评分趋势 | 衡量整体服务质量 |
| **处理效率** | 反馈处理时长 | 优化响应流程 |

### 5.2 分析报表（示例）

```
月度反馈分析报告
═══════════════════════════════════════

反馈总量：45条
├─ 功能建议：18条 (40%)
├─ 问题报告：12条 (27%)
├─ 体验反馈：10条 (22%)
└─ 其他：5条 (11%)

问题分布（按模块）：
├─ Mode-A 灵感构思：3条
├─ Mode-B 大纲架构：5条
├─ Mode-C 细纲规划：2条
├─ Mode-D 正文创作：4条
├─ Mode-E 诊断评估：8条 ← 重点关注
├─ Mode-F 包装运营：1条
└─ 其他：3条

满意度趋势：
4.2 → 4.3 → 4.1 → 4.4 → 4.5
（呈上升趋势）

本月重点改进：
1. Mode-E 诊断模块响应速度优化
2. 新增可视化辅助工具
3. 卡牌模式示例丰富

下月计划：
1. 继续优化诊断模块
2. 增加更多题材案例
3. 优化 SKILL.md 结构
```

## 七、持续改进机制

### 7.1 迭代周期

| 周期 | 活动 | 输出 |
|------|------|------|
| **每周** | 反馈汇总分析 | 周度反馈简报 |
| **每月** | 深度分析+优先级排序 | 月度改进计划 |
| **每季度** | 大版本规划 | 季度路线图 |

### 7.2 改进闭环

```
收集反馈 → 分析归类 → 制定方案 → 实施改进 → 验证效果 → 通知用户
    ↑                                                    │
    └──────────────── 持续迭代 ← 收集新反馈 ←────────────┘
```

## 八、用户参与激励

### 8.1 贡献者认可

- **月度贡献者**：每月反馈质量最高的用户
- **Bug猎人**：发现并报告重要Bug的用户
- **创意之星**：提出优秀功能建议的用户

### 8.2 反馈质量奖励

| 反馈质量 | 奖励 |
|---------|------|
| 优秀（详细+有建设性） | 优先响应 + 新版本预览 |
| 良好（较详细） | 快速响应 |
| 一般（基本描述） | 标准响应 |

## 附录：快速反馈入口

用户可在任何模式下输入以下指令快速反馈：

| 场景 | 指令 | 效果 |
|------|------|------|
| 任何时候 | `反馈` | 打开反馈表单 |
| 对当前输出不满 | `不满意` | 快速提交负面反馈 |
| 对当前输出满意 | `满意` | 快速提交正面反馈 |
| 发现错误 | `错误` | 快速报告问题 |
| 有改进建议 | `建议` | 快速提交建议 |
