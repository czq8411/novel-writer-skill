# 边界条件与异常处理

> 按需加载：遇到边界场景、异常、新手引导、模式切换时加载本文件。

## 边界定义

| 场景 | 处理方式 |
|------|---------|
| 用户要求超出网文范畴 | 礼貌拒绝，说明能力边界 |
| 设定冲突无法调和 | 启动Sequential Thinking推演最优解 |
| 文件读取失败 | 尝试模糊匹配，如失败则询问用户 |
| 记忆库信息缺失 | 基于已有信息推演，标注不确定性 |
| 用户要求违反核心铁律 | 拒绝执行，说明原因 |

## 异常处理流程

| 异常类型 | 检测方式 | 处理方式 |
|---------|---------|---------|
| 设定冲突 | 对比Memory与大纲 | 启动Sequential Thinking，输出A/B方案 |
| 文件错误 | Filesystem返回错误 | 模糊匹配→询问用户→创建新文件 |
| 逻辑断裂 | 章节间剧情不连贯 | 回溯前文→补充过渡→标注修改 |
| 人设崩塌 | 角色行为与设定不符 | 对比Memory→修正行为→说明原因 |

## 进阶场景适配

- **多线叙事**：主线占比≥60%，支线单次偏离≤2章
- **群像调度**：单章配角出场≤5人，核心配角每3章至少出场1次
- **POV视角切换**：单章≤3次，同一视角持续≥500字
- **倒叙/插叙**：倒叙单次≤当前章节30%篇幅
- **跨地图/跨位面**：跨地图间隔≥20章，新地图引入为渐近式

## 渐进式引导系统

### 用户等级识别

| 等级 | 特征 | 引导策略 | 指令集 |
|------|------|---------|--------|
| **L1 纯新手** | 首次创作，无任何写作经验 | 全程引导，每步解释"为什么"，提供更多示例和模板 | 简化版（15个核心指令） |

### L1纯新手引导路径（详细版）

#### Step 1: 灵感初现
```
🎯 目标：找到你的故事灵感
📝 操作：告诉我任何想法——一个场景、一个角色、一段对话，甚至只是一种感觉
💡 示例："我想写一个穿越到古代的厨师"、"一个能听懂动物说话的少年"
✨ 输出：完整创意方案（包含题材、主角、金手指、爽点设计）
```

#### Step 2: 框架搭建
```
🎯 目标：构建故事骨架
📝 操作：基于上一步的创意方案，系统自动生成大纲
💡 提示：大纲包含世界观、人物、情节线等7个核心模块
✨ 输出：7模块核心大纲
```

#### Step 3: 章节规划
```
🎯 目标：规划每章内容
📝 操作：输入"细纲"，系统会引导你规划章节
💡 提示：建议先规划前10章，后面可以边写边规划
✨ 输出：逐章细纲（包含核心事件、爽点、伏笔）
```

#### Step 4: 落笔成文
```
🎯 目标：写出第一个章节
📝 操作：输入"写正文"开始创作
💡 提示：第一章非常重要！要在开头抓住读者注意力
✨ 输出：完整章节内容+自动质量校验
```

#### Step 5: 精雕细琢
```
🎯 目标：提升写作质量
📝 操作：输入"诊断"检查章节质量
💡 提示：系统会指出问题并给出修改建议
✨ 输出：诊断报告+优化建议
```

#### Step 6: 包装上架
```
🎯 目标：完成作品包装
📝 操作：输入"书名"、"简介"、"标签"生成包装方案
💡 提示：好的书名和简介能吸引更多读者
✨ 输出：书名方案、简介、标签
```

### 新手专属辅助

**术语解释卡**：

| 术语 | 解释 | 示例 |
|------|------|------|
| 爽点 | 让读者感到"爽"的情节——主角展示能力、打脸反派、获得奖励 | 废柴主角逆袭打脸 |
| 钩子 | 章节结尾留下的悬念，让读者忍不住想看下一章 | "他转身离开，却没看到身后那双猩红的眼睛" |
| 伏笔 | 前面埋下的线索，后面会回收 | 第5章提到的玉佩，第50章成为关键道具 |
| 金手指 | 主角的特殊能力或优势 | 系统、老爷爷、前世记忆、特殊血脉 |
| 黄金三章 | 小说前三章，决定读者是否继续读 | 第一章钩子、第二章冲突、第三章小高潮 |
| 完读率 | 读者读完某一章的比例 | 越高越好，代表章节吸引力强 |

**常见错误预警**：

| 节点 | 常见错误 | 预警提示 |
|------|---------|---------|
| 大纲阶段 | 世界观过于宏大，人物过多 | "建议先聚焦1-2个核心人物，世界观可以边写边展开" |
| 细纲阶段 | 每章塞入过多事件 | "每章1个核心事件即可，给人物留出呼吸空间" |
| 正文阶段 | 开篇大段背景介绍 | "第一章第一句必须是具体画面或动作，比如：'刀光闪过'、'她猛地睁开眼睛'" |
| 诊断阶段 | 追求完美，反复修改 | "先完成再完美。前10章的主要目标是建立读者信任，不必过分纠结细节" |

**新手问答助手**：

> 新手模式下，随时输入"为什么"、"怎么写"、"什么是"等问题，会触发详细解释：

| 问题类型 | 示例 | 响应 |
|---------|------|------|
| 概念解释 | "什么是金手指？" | 详细解释+示例 |
| 写作技巧 | "怎么写好对话？" | 给出具体技法+示例 |
| 流程疑问 | "为什么要先写大纲？" | 解释重要性+好处 |

## 无缝模式切换

> 用户在任一模式中可直接切换到其他模式，无需退出当前模式。系统自动保存当前模式上下文，在新模式完成后可无缝返回。

### 切换指令

| 指令格式 | 示例 | 行为 |
|---------|------|------|
| `切换到[模式名]` | `切换到模式A` | 保存当前上下文，加载目标模式 |
| `[模式名] [操作]` | `模式E 诊断第5章` | 保存当前上下文，加载目标模式并执行操作 |
| `先[操作A]再[操作B]` | `先诊断第5章再继续写` | 顺序执行跨模式操作，中间自动切换 |
| `返回` / `继续` | `返回` | 恢复到切换前的模式和上下文 |

### 上下文保存与恢复

每次模式切换时，自动保存当前模式的状态快照：

```json
{
  "contextSnapshot": {
    "sourceMode": "模式C",
    "sourceState": {
      "currentChapter": 42,
      "lastAction": "完成第42章正文创作",
      "pendingAction": "等待用户确认是否继续第43章"
    },
    "projectPath": "./",
    "timestamp": "2026-05-12T14:30:00"
  }
}
```

**恢复流程**：
1. 目标模式任务完成 → 自动提示"是否返回[源模式]继续？"
2. 用户确认 → 从快照恢复源模式状态
3. 用户拒绝 → 保持在当前模式

### 跨模式操作链

支持将多个模式操作串联为一次指令：

```
用户输入：先诊断第5章，再根据诊断结果重写第5章，然后继续写第6章

执行链：
模式E（诊断第5章）→ 输出诊断报告 →
模式C（重写第5章）→ 应用诊断建议 →
模式C（写第6章）→ 正常创作流程
```

**操作链规则**：
- 链中每个操作完成后自动进入下一个
- 任一操作失败则暂停，询问用户是否继续
- 链长度不限，但建议不超过5个操作

### 模式切换保护

| 保护项 | 说明 |
|--------|------|
| **数据不丢失** | 切换前自动保存当前模式的创作进度到Memory |
| **上下文不混淆** | 每个模式的上下文独立存储，互不污染 |
| **可回退** | 支持多级回退（返回→再返回），最多保留5级上下文栈 |
| **冲突检测** | 若目标模式的操作可能影响源模式的数据，先提示用户确认 |

## 上下文预算管理

| 场景 | 优化策略 |
|------|---------|
| **大纲文件过大** | 按模块优先级分片读取——优先6.3（章节摘要）+5.2（主角成长）+7.3（伏笔表），其余按需 |
| **正文创作超过500章** | Memory写入改为"增量模式"——仅记录跨章节关键状态变更，单次≤500字 |
| **多章诊断** | 先输出综合评分+核心问题摘要（≤800字），详细分析按问题级别逐项展开 |

## 模式切换保护

| 切换方向 | 前置检查 | 阻断条件 | 引导操作 |
|---------|---------|---------|---------|
| 任意 → 模式C（正文） | 大纲文件存在且可读 | 大纲文件缺失 | 引导至模式B先生成大纲 |
| 任意 → 模式B-2（细纲） | 大纲文件存在且可读 | 大纲文件缺失 | 引导至模式B先生成大纲 |
| 任意 → 模式E（诊断） | 前3-5章细纲和正文存在 | 正文不足3章 | 放宽至已有章节数，标注样本不足风险 |
| 模式B → 模式B-2 | 大纲7模块基本完整 | 核心模块(1/5/6)缺失 | 提示补充缺失模块 |
