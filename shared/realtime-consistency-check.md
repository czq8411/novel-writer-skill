# 共享模块：实时一致性校验系统

> **按需加载**：每章正文生成前后自动加载执行
> **全局规则**：遵循主控文件第二章核心铁律
> **版本**：v21.1

---

## 一、系统概述

实时一致性校验系统用于在正文创作过程中实时检测和防止设定冲突、人物OOC、伏笔遗漏等问题，确保长篇创作的连贯性和一致性。

**核心目标**：
- 在问题发生前预警
- 在问题发生后立即修复
- 减少事后诊断的依赖

---

## 二、校验触发机制

### 2.1 自动触发点

| 触发时机 | 校验类型 | 说明 |
|---------|---------|------|
| **正文生成前** | 预检 | 检查细纲约束、人物状态、伏笔状态 |
| **正文生成中** | 实时检 | 每完成一个段落进行快速检查 |
| **正文生成后** | 完整检 | 全面一致性校验 |
| **每50章里程碑** | 深度检 | 回顾核心设定、梳理人物关系 |

### 2.2 手动触发

用户可随时输入以下指令进行校验：

| 指令 | 功能 |
|------|------|
| `一致性检查` | 执行完整一致性校验 |
| `设定检查` | 检查设定冲突 |
| `人物检查` | 检查人物一致性 |
| `伏笔检查` | 检查伏笔状态 |
| `时间线检查` | 检查时间线逻辑 |

## 三、校验维度与标准

### 3.1 设定一致性校验

#### 校验项

| 校验项 | 检查内容 | 通过标准 | 权重 |
|--------|---------|---------|:---:|
| **世界观设定** | 力量体系、地理、规则 | 无冲突 | 25% |
| **时间线** | 事件发生顺序、时间跨度 | 逻辑合理 | 20% |
| **物品属性** | 法宝、技能、物品功能 | 与设定一致 | 15% |
| **势力关系** | 宗门、家族、阵营关系 | 无矛盾 | 20% |
| **规则一致性** | 世界运行规则 | 前后一致 | 20% |

#### 设定冲突检测流程

```
提取当前章涉及的所有设定
    ↓
与Memory中的核心设定对比
    ↓
发现冲突？
    /        \
  是          否
  ↓           ↓
标记冲突    通过
  ↓
查询最近10章的设定使用
    ↓
判断是否为有意变更
    /        \
  是          否
  ↓           ↓
记录变更    生成修正建议
  ↓           ↓
提示用户    自动修正或标记
```

#### 设定冲突分级

| 级别 | 定义 | 处理方式 |
|:----:|------|---------|
| 🔴 P0-严重 | 核心设定冲突（如主角能力突变） | 强制修正，不通过 |
| 🟡 P1-重要 | 次要设定冲突（如配角名字错误） | 建议修正，可跳过 |
| 🟢 P2-轻微 | 细节不一致（如颜色描述差异） | 记录提醒，后续处理 |

### 3.2 人物一致性校验

#### 校验项

| 校验项 | 检查内容 | 通过标准 | 权重 |
|--------|---------|---------|:---:|
| **性格一致性** | 行为是否符合性格设定 | 无OOC | 30% |
| **能力一致性** | 技能使用是否符合境界 | 符合设定 | 20% |
| **关系一致性** | 人物关系是否符合当前状态 | 无矛盾 | 25% |
| **状态连续性** | 伤势、情绪、位置是否连续 | 合理过渡 | 25% |

#### OOC检测机制

```markdown
## OOC检测流程

### 步骤1：提取人物当前行为
- 动作：[列表]
- 对话：[列表]
- 决策：[列表]

### 步骤2：对比人物设定
- 核心性格：[设定]
- 当前行为是否符合？
  - 是 → 通过
  - 否 → 进入步骤3

### 步骤3：分析合理性
- 是否有外部因素导致？（如受伤、受威胁）
  - 是 → 检查外部因素是否已铺垫
    - 已铺垫 → 通过
    - 未铺垫 → 标记需要铺垫
  - 否 → 标记OOC

### 步骤4：生成建议
- 如果是合理变化 → 建议增加心理描写说明
- 如果是OOC → 提供符合性格的行为替代方案
```

### 3.3 伏笔一致性校验

#### 校验项

| 校验项 | 检查内容 | 通过标准 | 权重 |
|--------|---------|---------|:---:|
| **回收时机** | 是否在计划章节回收 | 按计划或提前 | 30% |
| **回收方式** | 回收是否符合预期 | 逻辑合理 | 25% |
| **埋设连续性** | 新伏笔是否与已有伏笔冲突 | 无冲突 | 25% |
| **超期检测** | 是否有超期未回收伏笔 | 无超期 | 20% |

#### 伏笔状态追踪

```markdown
## 伏笔状态表

| 编号 | 埋设章节 | 内容 | 计划回收 | 实际回收 | 状态 | 预警 |
|------|---------|------|---------|---------|------|------|
| F001 | 第5章 | 神秘玉佩 | 第50章 | - | 待回收 | 正常 |
| F002 | 第12章 | 老者身份 | 第80章 | - | 待回收 | 正常 |
| F003 | 第30章 | 宗门秘密 | 第100章 | - | 待回收 | ⚠️ 即将到期 |
| F004 | 第45章 | 敌人阴谋 | 第60章 | - | 待回收 | 🔴 已超期5章 |
```

#### 超期预警机制

| 预警级别 | 触发条件 | 自动响应 |
|:-------:|---------|---------|
| 🟢 提醒 | 距离计划回收≤5章 | 提示"伏笔即将到期" |
| 🟡 警告 | 超期≤10章 | 强制提示，建议尽快回收 |
| 🔴 严重 | 超期>10章 | 标记为遗漏，生成回收方案 |

### 3.4 时间线一致性校验

#### 校验项

| 校验项 | 检查内容 | 通过标准 |
|--------|---------|---------|
| **时间顺序** | 事件发生的先后顺序 | 逻辑合理 |
| **时间跨度** | 章节间的时间间隔 | 符合设定 |
| **时间标记** | 时间描述是否一致 | 无矛盾 |
| **季节/天气** | 环境时间是否连续 | 合理过渡 |

#### 时间线追踪

```markdown
## 时间线记录

### 故事时间轴
| 章节 | 故事时间 | 时间跨度 | 关键事件 |
|------|---------|---------|---------|
| 第1章 | 天元历1000年春 | - | 主角出场 |
| 第10章 | 天元历1000年夏 | +3个月 | 入门测试 |
| 第20章 | 天元历1000年秋 | +2个月 | 首次任务 |
| 第30章 | 天元历1000年冬 | +3个月 | 宗门大比 |

### 时间一致性检查
- 第31章提到"春天" → 🔴 冲突！第30章是冬天
- 修正建议：改为"虽然已是春天，但寒意未消"
```

## 四、校验执行流程

### 4.1 预检流程（生成前）

```
开始预检
    ↓
读取细纲约束
    ↓
读取人物状态（Memory）
    ↓
读取伏笔状态
    ↓
读取最近3章关键设定
    ↓
生成约束清单
    ↓
输出预检报告
    ↓
等待用户确认或修改
```

**预检报告格式**：
```markdown
## 一致性预检报告（第[X]章）

### 约束清单
- [x] 核心事件：[描述]
- [x] 人物状态：[主角状态]
- [x] 必须回收伏笔：[F001, F002]
- [x] 必须埋设伏笔：[F010]

### 风险提示
⚠️ 伏笔F003将在5章后到期，请确保本章或下章回收

### 建议
- 建议在本章提及F001的线索，为第50章回收做铺垫
- 主角境界即将突破，建议增加突破前的准备描写

**预检通过，可以开始创作**
```

### 4.2 实时检流程（生成中）

```
每完成一个段落
    ↓
快速检查：
  - 是否偏离核心事件？
  - 人物行为是否OOC？
  - 时间线是否合理？
    ↓
发现问题？
    /        \
  是          否
  ↓           ↓
立即提示    继续生成
  ↓
用户决策：
  - 修正
  - 忽略
  - 暂停
```

### 4.3 完整检流程（生成后）

```
正文生成完成
    ↓
执行完整一致性校验：
  - 设定一致性（25%）
  - 人物一致性（30%）
  - 伏笔一致性（25%）
  - 时间线一致性（20%）
    ↓
计算综合评分
    ↓
评分≥8分？
    /        \
  是          否
  ↓           ↓
通过        生成修正建议
  ↓           ↓
输出报告    自动修正或标记
```

**完整检报告格式**：
```markdown
## 一致性校验报告（第[X]章）

### 综合评分：8.5/10 ✅ 通过

### 各维度评分
| 维度 | 得分 | 状态 |
|------|:---:|:----:|
| 设定一致性 | 9/10 | ✅ |
| 人物一致性 | 8/10 | ✅ |
| 伏笔一致性 | 8/10 | ✅ |
| 时间线一致性 | 9/10 | ✅ |

### 发现的问题
🟢 P2-轻微：主角衣服颜色描述与前章不一致（前章蓝色，本章青色）
- 建议：统一为"青蓝色"

### 修正建议
1. [可选] 统一主角衣服颜色描述

### 总体评价
本章一致性良好，轻微问题不影响阅读，可选择性修正。
```

## 五、自动修正机制

### 5.1 修正优先级

| 优先级 | 问题类型 | 自动修正 | 用户确认 |
|:------:|---------|:-------:|:-------:|
| P0 | 核心设定冲突 | ✅ | 必须 |
| P1 | 人物OOC | ⚠️ | 建议 |
| P1 | 伏笔遗漏 | ✅ | 可选 |
| P2 | 细节不一致 | ❌ | 可选 |

### 5.2 修正流程

```
发现问题
    ↓
判断优先级
    ↓
P0/P1？
    /        \
  是          否（P2）
  ↓           ↓
生成修正方案  记录到待修复清单
    ↓           ↓
自动执行修正  用户选择是否修复
    ↓
验证修正结果
    ↓
通过？
    /        \
  是          否
  ↓           ↓
完成        标记人工处理
```

### 5.3 修正策略

| 问题类型 | 修正策略 |
|---------|---------|
| 设定冲突 | 回退到最近一致状态，重新生成冲突部分 |
| 人物OOC | 根据性格设定重写行为/对话 |
| 伏笔遗漏 | 在合适位置插入伏笔回收/埋设 |
| 时间线错误 | 调整时间描述，确保逻辑合理 |
